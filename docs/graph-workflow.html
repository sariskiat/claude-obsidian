Stage by stage

1
Your input · external skill

Extract the paper

A research PDF is turned into structured knowledge (title, sections, concepts, relationships) by the paper-scholar / paper-master skills. These live outside claude-obsidian — they are the front door that produces clean, machine-readable paper content.

/paper-scholar extract paper.pdf
2
Semi-automated today · native automation is the next step

Ingest into the claim graph

Extracted papers become the graph's atoms: entities (typed: Concept / Method / Artifact / Task / Measure / Source) and claims — typed subject —predicate→ object triples carrying polarity (asserts/refutes), strength, and a support count, each anchored to a verbatim quote.

Today there are two real ways new papers land as markdown under wiki/graph/:

  • Hand-author a papers/, entities/, claims/ markdown file set following the contract in wiki/graph/SCHEMA.md, then run the build.
  • Bulk import from an existing sqlite graph via graph-export.py — this is how the current 98-paper corpus got here.
SCHEMA.md = the frontmatter contract
3
Native · the source of truth

wiki/graph/ — plain markdown you own

This directory is the truth. Git-tracked, Obsidian-readable, hand-fixable. If the extractor was noisy, you edit the markdown — the quote and triple are right there to judge.

papers/<slug>.md · entities/<name>__e<id>.md · claims/c<id>.md · _graph/ aux tables · graph-export.json portable snapshot.

Kept separate on purpose: wiki/graph/entities/ (structured, typed) is distinct from wiki/entities/ (prose pages). Merging them would flatten the claim layer back into a plain organizer — the one thing this design exists to prevent.
4
Native · automated

Build the derived index

The markdown is compiled into a throwaway sqlite index. It is gitignored and can be deleted any time — it is rebuilt from the markdown, never the other way around. Entity rollups use one shared, cycle-safe root() helper (never the old single-hop COALESCE).

uv run python scripts/graph-build.py wiki/graph .vault-meta/graph/graph.db # → papers 98 · entities 1444 · claims 1052 · aliases 834 …
5
Native · automated

Analyze — gaps & duplicate resolution

Over the derived index you run the analytics the page wiki could never do:

  • Gap scan — five species: frontier, debate (asserts vs refutes on one triple), replication (single-support results to confirm), coverage (thin entities), white-space (disconnected community pairs).
  • Entity resolution — proposes duplicate merges (exact-name, then embedding / token-Jaccard similarity). Proposes only; you confirm.
uv run python scripts/graph-gaps.py --top 20 # research gaps uv run python scripts/graph-resolve.py --json # merge proposals
6
Native · the compounding loop

Edit → rebuild → re-analyze

Fix a wrong claim or merge two entities in the markdown, rebuild, and the gap counts update. Because truth is markdown in git, every correction is durable and versioned — the graph compounds instead of decaying.

edit wiki/graph/*.md→ graph-build→ graph-gaps

The one gap in the loop: getting a brand-new paper into the graph is not yet a single command — graph-ingest.py is specified in SCHEMA.md but not implemented. Until it exists, new papers enter by hand-authoring markdown to the contract, or by bulk-importing an existing sqlite graph. Everything from “markdown” onward is fully automated and verified.

Why a claim graph and not just pages

The moat · claim layer

wiki/graph/ — claims

A typed, signed, quoted proposition is the atom. Only this can answer “Paper A asserts X, Paper B refutes X”, count replications, and surface the five gap species.

  • subject —predicate→ object
  • polarity · strength · support
  • verbatim quote = provenance wall
The organizer · page layer

wiki/ — prose pages

The existing /wiki family: readable pages, links, hybrid retrieval, methodology modes. Excellent at recall and narrative — but it cannot represent contested claims.

  • pages, wikilinks, hot/index cache
  • BM25 + rerank retrieval
  • LYT / PARA / Zettelkasten modes

Command quick reference

# rebuild the index from markdown (safe any time; the db is disposable) uv run python scripts/graph-build.py wiki/graph .vault-meta/graph/graph.db # scan for research gaps (human report, or JSON) uv run python scripts/graph-gaps.py --top 20 uv run python scripts/graph-gaps.py --json --top 50 # propose duplicate-entity merges (you confirm each) uv run python scripts/graph-resolve.py --json # bulk-import an existing sqlite graph into wiki/graph/ markdown uv run python scripts/graph-export.py <source.db> wiki/graph # run the test suite (39 tests) make test-graph
